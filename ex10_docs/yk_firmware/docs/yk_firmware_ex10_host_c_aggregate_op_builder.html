Ex10Ops Aggregate Op Builder¶

The aggregate op builder adds helpers in the process of building a buffer to utilize with the aggregate op. The aggregate op can be used to perform a combination of writes, running ops, etc (A list of supported instructions can be found bellow). The buffer builder can be utilized at many different levels. A user may choose to insert a single instruction, or they may choose to insert common combinations which can incluse some setup registers followed by an op.

struct Ex10AggregateOpBuilder¶

Interface to help with building the Ex10 aggregate op buffer.

bool (*append_instruction)(const struct AggregateOpInstruction op_instruction, struct ByteSpan *agg_op_span)¶

Appends the passed instruction into the passed span.

The function looks at the current instruction and knows how to add it to the passed span. The length and data fields of the span are updated with the new data. NOTE: If the span length does not change, nothing was added.

Param op_instruction

The current instruction being added

Param agg_op_span

The span which we are adding to. The length and data field will be updated

Param return

Returns false if there isn’t enough room to add the new instruction, else it returns true.

void (*clear_buffer)(void)¶

This clears the device side aggregate op buffer by writing all 0s to the register which contains the buffer.

bool (*set_buffer)(struct ByteSpan *agg_op_span)¶

Sets the passed byte span data into the device.

There is a buffer register used in an Ex10 device to read from in executing the aggregate op. This register is set with the data in the byte span according to the length dictated in the byte span.

ssize_t (*get_instruction_from_index)(size_t index, struct ByteSpan *agg_op_span, struct AggregateOpInstruction *instruction_at_index)¶

Takes in the index of interest and finds the associated instruction data if it exists.

Param index

The index to check.

Param agg_op_span

The current buffer data to check through.

Param instruction_at_index

The instruction data at the given index. If the index is not found, this will be garbage data.

Param return

Returns the base 0 number of the instruction corresponding to the passed index. This is the instruction number, not the index. Will return a -1 if the index is not a valid instruction.

void (*print_buffer)(struct ByteSpan *agg_op_span)¶

Scans through the passed span and prints the contained instructions.

This means it decodes the span as it reads through and prints what the sequential order of instructions within. NOTE: This will not print logical flow AKA it will not follow jump logic.

bool (*append_reg_write)(struct RegisterInfo const *const reg_info, struct ConstByteSpan const *data_to_write, struct ByteSpan *agg_op_span)¶

Appends a register write to the passed in aggregate buffer.

Note

When used as an ex10 command, writes only require a single instruction code and can be followed by multiple write address/length operations. In the aggregate op, we need to do write commands one at a time. This means we can only write to one register per write instruction.

Param data_to_write

The data to add.

Param agg_op_span

The local copy of the aggregate buffer.

Param return

Returns false if a buffer overflow occurs in adding to the buffer

bool (*append_insert_fifo_event)(const bool trigger_irq, const struct EventFifoPacket *event_packet, struct ByteSpan *agg_op_span)¶

See

ops layer append_insert_fifo_event() call

Note

Works the same as the normal Ex10-api command with one exception. Normally, the insert fifo command can be used with a NULL packet by sending nothing more than the command id and the trigger irq. The aggregate op can not find the commmand length if the packet is NULL. If you do not wish to send out a packet, use a packet header which dictates static data length and 0 dynamic data length. This empty packet will not be sent out, but an irq can still be generated.
