Impinj Reader Chip EventFifo Packets¶

The Impinj reader chip sends asynchronous data to the host using the ReadFifo command. This command allows the host to read packets from the EventFifo. Within the EventFifo are packets which are described in the tables and figures below.

There are a few rules which apply to the EventFifo packets:

  • EventFifo packets are always multiples of 32-bit words in length. When a packet, as shown below, does not end on a 32-bit boundary, it will be zero-padded to the next 32-bit boundary.

  • Each packet begins with a packet header. The packet header is 8 bytes in length, occupying bytes 0 through 8 of the packet data. Each packet header is format consists of the following bytes:

    • The packet_length field, located at byte offset 0, indicates the length of the entire packet in 32-bit words (not bytes), and includes the packet header.

    • The EventFifo packet_type is located at byte offset 1.

    • A 16-bit SHA is located at byte offsets 2 and 3. This SHA value changes with each release of firmware.

    • The us_counter, located at byte offsets 4 through 7, is the time when the packet was inserted into the reader chip EventFifo. The us_counter value starts at the value zero when the reader chip device is reset.

  • All fields are sent with the least significant octet first. i.e. little endian order.

EventFifo packets may contain dynamic data. Dynamic data is the variable length portion of the payload which is not shown in the payloads which follow.

To determine the length of the dynamic portion of the packet the following formula applies:

dynamic_data_length = packet_length - static_packet_length

Where the static_packet_length is the fixed packet length as shown in the packets below, and includes the packet header. The dynamic data continues beyond the static packet length until the header field packet_length is satisfied. Again, all packet length value units are 32-bit words.

Event tag_read¶

Indicates a tag singulation. This packet contains dynamic data.

Fields:

Offset

Name

Description
0 packet_length

The length of the entire packet in 32-bit words.
1 packet_type

The EventFifo packet type. @see enum EventPacketType
2 sha

A SHA which is generated based on the EventFifo format.
4 us_counter

Packet timestamp, in microseconds, from the Impinj reader chip.
8 rssi

ADC value of the RSSI of the tag reponse
10 rf_phase_begin

The RF phase of the tag response at the beginning of the packet. This is a 12-bit unsigned value with a resolution of 0.087 degrees per bit, representing the full range of phase.
12 rf_phase_end

The RF phase of the tag response at the end of the packet. This is a 12-bit unsigned value with a resolution of 0.087 degrees per bit, representing the full range of phase.
14 halted_handle

If the modem has been instructed to halt on all tags, this is the handle that was returned with the Req_RN Gen2 command
16 type

The data contained in the dynamic data

    0x1 : epc : The data contains the PC-EPC-CRC of a normal tag read.

    0x2 : epc_with_tid : The data contains a PC-EPC-CRC and then the Gen2 Read of the TID

    0x3 : epc_with_fast_id_tid : The data contains the fast_id response from a tag that supports fast_id. The format could be either PC-EPC-TID-CRC or PC-EPC-CRC-TID-CRC.
17 tid_offset

The offset, in bytes, from the beginning of dynamic data in the tag_read packet to the TID if the TID was received.

If no TID was received then the tid_offset value is 0.

  • For epc type, this will be set to 0.

  • For epc_with_tid type, this will be the length of PC+EPC or PC+EPC+CRC.

  • For epc_with_fast_id_tid type, this will be the length of the EPC received, meaning the length of PC+EPC+CRC.
18 halted_on_tag

If true, the modem is currently halted on a tag.
19 packet_rfu

Padding byte with value 0x00

Event gen2_transaction¶

The result of a user requested Gen2 transaction

Fields:

Offset

Name

Description
0 packet_length

The length of the entire packet in 32-bit words.
1 packet_type

The EventFifo packet type. @see enum EventPacketType
2 sha

A SHA which is generated based on the EventFifo format.
4 us_counter

Packet timestamp, in microseconds, from the Impinj reader chip.
8 transaction_id

When the user specifies a transaction to perform, one of the fields is a transaction id. The only purpose this serves is to allow the user to more easily synchronize requests with results. The modem does not use this for anything internally.
9 status

The status of the Gen2 transaction that the modem observed. Note that in the case of a Gen2 error for packets that contain the header bit, this status will indicate ok and the host should look at the payload bits to examine whether the header bit indicates a Gen2 error, and if so, the error code will be in the payload buffer as described in the Gen2 tag reply documentation.

    0x1 : ok : The transaction completed with valid data.

    0x2 : bad_crc : When the CRC-16 was checked, it was found to be incorrect. If the transaction was not supposed to have a CRC-16 on the end of the packet, this status code can still indicate that the reception was ok.

    0x3 : no_reply : The modem was not able to detect a response from the tag or was not configured to expect a response.

    0x4 : invalid_reply_type : The reply type specified in the Gen2TxnControls register is either invalid or unsupported.

    0x5 : cover_code_failed : The cover code was requested but the response to the ReqRn failed CRC.
10 valid_path_metrics

This boolean value indicates if the path metrics exceeded the minimum thresholds.
11 packet_rfu

Padding byte with value 0x00
12 rf_phase_begin

The RF phase of the tag response near the beginning of the packet. This is a 12-bit unsigned value with a resolution of 0.087 degrees per bit, representing the full range of phase.
14 rf_phase_end

The rf phase of the tag response near the end of the packet. This is a 12-bit unsigned value with a resolution of 0.087 degrees per bit, representing the full range of phase.
16 rssi

ADC value of the RSSI of the tag reponse.
18 num_bits

Number of bits in the response

A 16 bit value which identifies the Impinj reader chip SKU.

  • 0x0510: E510

  • 0x0310: E310
10 reset_reason

A bitfield of reset reasons of which there are 4. If none are set then it was a normal power on reset (POR)

  • Bit 0: System reset (internal reset signal)

  • Bit 1: Watchdog timeout

  • Bit 2: Cortex lockup

  • Bit 3: External reset (the RESET_N pin was set low)

This field contains the crash info conditional value that indicates the crash info status

  • 0: Unknown (illegal state)

  • 1: Reset normal

  • 2: Crash

  • 3: Fault

Contains summary data of what was run in the aggregate op.

Fields:

Offset

Name

Description
0 packet_length

1 packet_type

The EventFifo packet type. @see enum EventPacketType

A SHA which is generated based on the EventFifo format.
4 us_counter

Packet timestamp, in microseconds, from the Impinj reader chip.
8 op_run_count

The number of op runs which were executed.
10 write_count

The number of write commands which were executed.
12 insert_fifo_count

The number of insert fifo events which were executed.
14 final_buffer_byte_index

The aggregate op parses through the aggregate op buffer. This index shows the final byte index of the op within the buffer. This means it is the bytes at which the parsing stopped, whether that is due to proper completion or error. If the last byte was the end instruction code, the index will show the byte of that instruction since it caused the exit. If there is an error in a given command, the index will point to the beginning of the instruction that caused the error.
16 total_jump_count

The summed count of all jumps executed.
18 last_inner_op_run

In the aggregate op, if an op in the buffer fails, the op will return with the appropriate error code. This field can be used to show which op failed in the buffer.
19 last_inner_op_error

If the most recent op run within the aggregate op fails, the error code of this failure will be placed here before the op exits.
