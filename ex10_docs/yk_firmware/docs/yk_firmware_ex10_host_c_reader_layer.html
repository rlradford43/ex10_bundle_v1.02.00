The Ex10Reader interface provides the highest level of abstraction, yet is still a full featured interface for performing RAIN RFID tasks. The Impinj Reader Chip Host API is described in this section.

void (*packet_remove)(void)¶

Delete the packet at the front of the packet queue.

Note

There is a limited amount of room for packets in the packet queue, so packets should be discarded regularly using remove_packet() to ensure room for new packets.

Note

The packet must only be removed once it is not longer in use. packet_remove() is similar to calling free() on allocated memory. Once the packet has been removed its packet pointer obtained through packet_peek() is invalid.

Note

Calling packet_free() allows for new packets to be read from the Ex10. If packet_remove() is not called then reading will stall and the available buffer space will have been used up.

bool (*packets_available)(void)¶

Note

Return

bool true if packets are available for reading. false if there are no packets available for reading.

int16_t (*get_listen_before_talk_rssi)(uint8_t antenna, uint32_t frequency_khz, int32_t lbt_offset, uint8_t rssi_count, bool override_used)¶

Runs the LBT op, retrieves the RSSI measurement from the event FIFO, and runs the value through compensation.

Param antenna

The antenna to transmit on.

Param frequency_khz

If a frequency is specified here, this forces the stack to always ramp up to the same frequency. If 0, the region specific jump table will be used.

Param lbt_offset

The offset frequency for measuring the RSSI.

Param rssi_count

The integration count for measuring RSSI.

Param override_used

Determines whether or not to use the Rx gain override setting in the LBT control register. This is normally false, which means the default LBT-specific settings are used in RSSI measurement and compensation. These default settings override the RxGainControlFields in the RxGainControl register. If the override is set, the user-specified settings in the RxGainControl register will be used during measurement and compensation. Note that these user settings must be set before this function call.

Return

The compensated RSSI value in cdBm. In case of any error which causes an issue in reading the RSSI, RSSI_INVALID is returned.

bool (*get_return_loss_threshold_exceeded)(uint16_t return_loss_cdb, uint16_t tx_power_cdbm, uint32_t frequency_khz, int16_t max_margin)¶

Determines if the return loss threshold was exceeded.

Using user passed parameters and board specific losses, this function checks if the current measurement on the reverse power detectors is above the allowed value.

Param return_loss

The return loss at the board output in cdB.

Param tx_power_dbm

Target transmit power level in cdbm.

Param frequency_khz

The frequency currently ramped to which is the same frequency used for the reverse power detector measurement.

Param max_margin

Margin to ensure we detect a poor return loss condition. Note that a higher max_margin creates a lower tolerance for the reverse power detector.

Return

Whether the threshold was exceeded. A true means that the reverse power detector measured higher than expected, and a false means we are within expectations.

Reverse Power Threshold Detection¶

The return loss detection mechanism in the reader layer allows a way to check the current return loss and compare against a threshold.

This check can be called through the reader function get_return_loss_threshold_exceeded. This allows the user to utilize this functionality at any time they desire. This can be helpful in detecting if the current antenna has been disconnected, if the antenna is damaged, or if something unexpected has happened in the TX/RX pathway. Note that this function can only be called when an op is not running, so be sure to wait for completion of previous ops before calling.

A sequence diagram can be seen for the reverse power threshold detection call below…

In addition to calling this check through the reader layer when desired, there is a call to check the reverse power as a callback after each rampup. If the user uses the reader layer for ramp up, the integrated ramp-up process uses a reader layer callback function which get registered to the post_rampup_callack callback in the ops layer. The user can alter this callback to change the parameters/threshold to this check based on customer board specifications. If desired, this check can be removed entirely from the process.
