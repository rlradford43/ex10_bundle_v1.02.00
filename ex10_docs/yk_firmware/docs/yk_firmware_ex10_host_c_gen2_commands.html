Gen2 Commands¶

This gen2 commands file allows for organization and interpreting of different gen2 commands. This file therefore has knowledge about all different gen2 commands, what their individual fields consist of, etc. This file is then used to encode and decode gen2 commands and responses.

struct BitSpan¶

Defines a mutable location of contiguous memory, organized as bits.

Each byte in data is utilized, therefore 8 bits from length exist in each byte. The bits and bytes are organized LSB first. This means that bit 0 of byte 0 is the first bit, bit 1 of byte 0 is next, etc.

struct Ex10Gen2Commands¶

Gen2 commands encoder/decoder interface.

Public Members

bool (*encode_gen2_command)(const struct Gen2CommandSpec *cmd_spec, struct BitSpan *encoded_command)¶

Encodes a Gen2 command for use in the Gen2TxBuffer.

load_mask = 0x10 load_bits = 4 bits_loaded = 0b0001

Note

For select mask only: Everything is loaded in left-justified. This means the bytes are loaded in via lsb, but the bits are loaded in left to right as well, making them bitwise msb. This was done for reader clarity to make it look like a bitstream EX: load_mask = 0xa2f5 load_bits = 12 bits_loaded = 0xa2f

Parameters

  • cmd_spec – A struct containing: 1) command - The command type as an enum Gen2Command 2) args - A pointer to an arg struct for the specified command, which contains values for the fields in the CommandEncoder table.

  • encoded_command – [out] A bit span containing info about the encoded command. This contains the length of the command and data for the left-justified Gen2 command.

  • [return] – Returns false if the command type is invalid, otherwise returns true.

Gen2 Tx Command Manager¶

The Gen2 commands module revolves around a separate standalone buffer in the SDK with its own helpers. This means there is a local buffer in the SDK separate from the register in the Ex10 device. The Gen2 Tx Manager is responsible for modifying the local buffer and updating the device accordingly.

Gen2 Tx Command Manager Usage¶

Step 1 – Steup In working with this buffer, rather than the command structures being intertwined with the ops, one begins by retrieving the gen2 sequence helper.

struct Ex10Gen2TxCommandManager const* g2tcm =

get_gen2_tx_command_manager(void);

The buffer builder has an init function which is called in the SDK reader layer, but should be called if not utilizing that layer. This invalidates all commands in the local buffer, device buffer, and properly allocates the backing memory of the structures in the local gen2 command storage. The buffer clearing is already done in the reader initialization through…

void init (void);

This clears bot the local and device side buffers. Both of these things can be done individually of course through the following…

void clear_sequence(void); void clear_local_sequence (void);

This is to ensure there is nothing leftover in the sequence storage that we did not intend to use. There is not a need to utilize this every time you want to add something in. There is, however, a max of 10 commands. If you still want previous commands, but are running close on the command limit, you can always clear a single index of the sequence through the following…

struct Gen2TxCommandManagerError clear_command_in_local_sequence(uint8_t clear_idx);

Step 2 – Error Management You will notice that a number of the functions used in the sequence manager return the following structure…

struct Gen2TxCommandManagerError

This is used to inform the user what happened within the api. The main piece to check is the ‘error_occurred’ field. If this boolean is set to true, you will want to check the error field to find what specifically happened. There is also a ‘current_index’ field which gets set to inform the user which index is being operated on if any. An example would be if you are clearing out a command using the previously mentioned…

struct Gen2TxCommandManagerError clear_command_in_local_sequence(uint8_t clear_idx);

If successful, the ‘current_index’ field will be set with the index which was cleared. If not, the field dictates the index at which the error occurred.

Step 3 – Adding commands The buffer is now ready to accept new commands. There are 2 ways to add a new command to the local buffer. You can call one of the following…

struct Gen2TxCommandManagerError (*append_encoded_command)(

const struct BitSpan* tx_buffer, uint8_t transaction_id);

struct Gen2TxCommandManagerError (*encode_and_append_command)(

struct Gen2CommandSpec* cmd_spec, uint8_t transaction_id);

This gives you the option of adding in a pre-encoded command, or adding in a command specifier which will be encoded for you. If you go the route of adding in a command spec, the api will encode for you then store a copy of the command specifications and the encoded command for ease of use. The other API function is to allow pre-encoding of the command, thus giving the user flexibility in creation of commands. This will still allow you to utilize the gen2 encoder by yourself if you need to add in something custom…

bool (encode_gen2_command)(const struct Gen2CommandSpec cmd_spec,

struct BitSpan* encoded_command);

To check what has been added to the buffer at any time, you may use the API… struct TxCommandInfo* (*get_local_sequence)(void);

Step 4 – Ex10 device side management When the SDK buffer contains all the commands of interest, it is time to write it to the device. One uses the API command…

struct Gen2TxCommandManagerError (*write_sequence)(void);

This write the sequence of commands to the device Gen2Tx register. The api also takes care of updating all of other registers which are used to control the Gen2Tx buffer. This includes the length registers, offsets, transaction ids, and txn controls.

Registers updated: Gen2Offsets, Gen2Lengths, Gen2TransactionIds, Gen2TxnControls, Gen2TxBuffer The Gen2Offsets register tells the device where in the Gen2TxBuffer register the command starts. The Gen2Lengths register tells the device how long the command at that offset is.

Note that since the SDK API commands were used to build the local buffer, the SDK is capable of creating the offsets and lengths itself. The Gen2TransactionIds are an indication to the user of which command is sent when. When a command is sent out of the transmitter, the event fifo packet notifies the user what happened along with this transaction ID. The Gen2TxnControls controls the device as to how each halted or auto access command is sent. These controls include what gen2 response type to expect, the rx length to look for, etc. The writing of the sequence to the device, however, does not touch the enables for the commands. This is done separately, though it can be done before or after updating these registers.

Example: For clarity on how a command is stored in the device… If index is enabled - Gen2Offsets[index] = offset_used Gen2Lengths[index] = length_used command_start = Gen2TxBuffer[offset_used] command_end = Gen2TxBuffer[offset_used + length_used]

Step 5 – Enabling commands With your sequence in the device, you now want to set the enable registers to let the device know which commands to run. There are three different enable registers on the device, which enable sending commands from the sequence under differernt situation. The interface allows you to load in 10 commands, but only enable a given subset at a time dependent on tag, timing, etc. The three different registers are…

Registers: Gen2SelectEnable, Gen2AccessEnable, Gen2AutoAccessEnable

General example: 1. Commands 1 through 5 are loaded into the device. 2. Gen2SelectEnable sets index 2 to be enabled, indicating that ther second index in the sequence is a select command. 3. Gen2AccessEnable sets index 1 and 3 to be enabled to utilize during a halted scenario. 4. Gen2AutoAccessEnable sets index 3 and 4 to be enabled for the auto access situation. This indicates that index 3 which is already used for halted, should also be used for auto access. 5. The Gen2AccessEnable register is update to enable only index 5 since we want to change which command in the sequence we send. No update to the sequence needs to be made since the commands are pre-loaded.

There are 3 separate functions used in updating these 3 enable registers…

struct Gen2TxCommandManagerError (write_select_enables)( bool const select_enables, uint8_t size); struct Gen2TxCommandManagerError (write_halted_enables)( bool const access_enables, uint8_t size); struct Gen2TxCommandManagerError (write_auto_access_enables)( bool const auto_access_enables, uint8_t size);

The bool pointers indicate that a user is meant to pass in an array of bools indicating which indices in the sequence to enable. The max of this array is the the max number of commands. If you exceed this, an eror will be returned. If you pass a size less than this, the rest of the bits up to the max will be filled with 0s.

Step 6 – Running commands With the command set up and enabled, all that’s left is to run them at the appropriate time. There are different ways, however, to run different commands in the sequence.

Select - To run the select op, there is a call in the ops layer…

void send_select(void)

This will execute the op assuming no other ops are running. There is another way to utilize this through the reader layer, however. If you are using the reader level inventory command, there is a parameter “send_selects”. This will also execute the select op after ramping up.

Access - To run a command during inventory, there are two methods: 1. If you are halted on a tag, you may set the ‘Go’ bit in the HaltedControl register. This tells the device to execute the command sequence as dictated by the previously set halted enables. This ‘Go’ bit can be set from the ops layer command…

void send_gen2_halted_sequence(void)

2. The other option is through the auto access feature. If you enabled any auto access commands they will be sent automatically when a tag is found.
