Impinj Reader Chip Messages¶

The Impinj reader chip and host communicate via SPI transactions. A SPI transaction is either a command message or a command message followed by a response message. Note that only some commands have responses. A command message consists of a command code byte followed by a payload. A response message consists of a reponse code followed by a payload of the requested size. All response sizes are known prior to the host receiving them, there is never a situation where a host must examine the received data to determine the length of the transfer. Command Details¶

Details about the command formats follows:

Read¶

The read command allows for reading segments of data (sequences of bytes) from the address map. You can bundle several reads together into a single command for efficiency purposes. It is up to the host to unpack and map the response bytes to the requested read segments.

Requested segments may span multiple adjacent addresses.

Reading invalid or write-only memory will result in padding bytes being returned in the response. In this manner the response buffer contains the expected number of bytes requested even when reading invalid memory.

A read command is malformed when the command buffer contains a number of bytes N, where (N - 1) % 4 is not equal to zero. Since the zeroeth byte in the command buffer is the read command identifier each subsequent set of bytes must specify the (address, length) pair for the read request. Sending a malformed read command will result in the error ‘MalformedCommand` response code.

Malformed read commands cannot be parsed properly. The length and/or address of the read command cannot be determined. In this situation the Impinj reader chip response code will be set to ‘MalformedCommand’.

A read command which requests a response buffer longer than the available response buffer will result in the error ‘ResponseOverflow’ in the response code field. No other valid data bytes will be present in the response buffer.

When a read command error occurs the response code byte will be the only valid field in the response buffer. All subsequent bytes clocked out via the serial interface will be invalid data.

Write¶

The write command allows for writing segments of data (sequences of bytes) into the address map. You can bundle several writes together into a single command for efficiency purposes.

Requested segments may span multiple adjacent addresses. Writing to undefined or read-only addresses are silently ignored.

A write command is malformed when the command buffer does not contain suffcient data content to complete the write operation. One example of an malformed write command is a write command with fewer payload bytes than are specified in the write request length field.

Sending a malformed write command will result in the error ‘MalformedCommand`. In the event of a write command error the write command will be rejected and no data will be written.

The write command has no response. To determine the outcome of a write command, the CommandResult address should be read.

ContinueUpload¶

The ContinueUpload command is the second of the three upload commands used when in the bootloader to upload new images into the chip. Because of the 2kB limit on payload size, images will often need to be broken out into multiple chunks for uploading.

An upload must be in progress as started by a StartUpload command. If this is not the case, the command will be interpreted as an invalid command.

CompleteUpload¶

The CompleteUpload command is the last of the three upload commands used when in the bootloader to upload new images to the Impinj reader chip.

An upload must be in progress as started by a StartUpload command. If this is not the case, the command will be interpreted as an invalid command.

If the image destination is the FLASH memory then the image validity markers are updated based on validating the signature with the public key.

The CompleteUpload command has no response. To determine the outcome of the command, read from the CommandResult address.

ReValidateMainImage¶

The ReValidateMainImage command allows a host to check whether the integrity of the main image held in FLASH has been compromised. When this command is run the signature of the main image is validated against its public key. If validation fails then the validity marker ‘MarkedInvalid’ will be set to true and the main image will not be allowed to execute until a new main image is successfully uploaded and validated using the [Start|Continue|Complete]Upload commands. Note that if the main flash image validity markers are set to an image invalid state, running this command never ‘repairs’ the validity markers to a valid state whereby the main image can be executed. The image validity markers are only set to a valid state within the CompleteUpload command.

Image headers include a field that can constrain an image to a specific Impinj reader chip, specifically using the chip’s unique serial number. The ReValidateMainImage command additionally checks that the serial numbers match for node-locked images. This check is unlikely to fail as the image would not have been sucessfully uploaded in the first place if the node-lock serial number didn’t match.

WriteInfoPage¶

If allowed and the CRC passes, erase the specified page and write the contents of this command into the specified info page. This command has an additional CRC field because misconfigurations in the info pages can have catastrophic consequences. The Gen2 16-bit CRC was chosen because it is required by the firmware to implement other Gen2 specific features and can be reused here.

InsertFifoEvent¶

The InsertFifoEvent command allows the host to inject an arbitrary event into the EventFifo. The insertion point of the event is the end of the EventFifo.

This command may also be used to force an EventFifo threshold interrupt regardless of if it is sending an EventFifo packet. The first byte is always the command id byte, and the second is the trigger_irq byte. If set to 1, this byte will trigger the irq. A user may choose to send these 2 bytes alone with no packet appended.

If the user does insert an EventFifo packet, the event format consists of the format described in the Impinj Reader Chip EventFifo Packets section. Any packet may be inserted by the host. The host must provide the packet header (length, packet type, sha, us_counter) and the event packet contents. The packet contents may or may not include dynamic data.

The user has a variety of options when inserting an event fifo command, but there are certain requirements in order to interpret the command properly.

If the EventFifo packet does not conform to the following formatting rules, it will be rejected as malformed (CommandMalformed).

  1. There must be enough bytes to fill at least the packet header (8).

  2. The EventFifo packet contents must be a multiple of 4 bytes in length.

  3. The denoted packet length in the header must match the payload length.
